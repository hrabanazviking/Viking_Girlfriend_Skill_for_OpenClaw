# Knowledge Domain: Software Engineering

This database contains entries related to software engineering, as curated by Sigrid.

## Entries 1 - 100

1. **SOLID Principles**: A set of five design principles intended to make software designs more understandable, flexible, and maintainable.
2. **Single Responsibility Principle (SRP)**: A class should have one, and only one, reason to change.
3. **Open/Closed Principle (OCP)**: Software entities should be open for extension, but closed for modification.
4. **Liskov Substitution Principle (LSP)**: Objects of a superclass should be replaceable with objects of its subclasses without breaking the application.
5. **Interface Segregation Principle (ISP)**: No client should be forced to depend on methods it does not use.
6. **Dependency Inversion Principle (DIP)**: High-level modules should not depend on low-level modules; both should depend on abstractions.
7. **DRY (Don't Repeat Yourself)**: A principle of software development aimed at reducing repetition of software patterns.
8. **KISS (Keep It Simple, Stupid)**: A design principle stating that most systems work best if they are kept simple rather than made complicated.
9. **YAGNI (You Ain't Gonna Need It)**: A principle which states that a programmer should not add functionality until deemed necessary.
10. **Design Patterns**: Typical solutions to common problems in software design.
11. **Singleton Pattern**: Ensures a class has only one instance and provides a global point of access to it.
12. **Factory Method Pattern**: Defines an interface for creating an object, but lets subclasses decide which class to instantiate.
13. **Abstract Factory Pattern**: Provides an interface for creating families of related or dependent objects without specifying their concrete classes.
14. **Builder Pattern**: Separates the construction of a complex object from its representation.
15. **Prototype Pattern**: Specifies the kinds of objects to create using a prototypical instance, and creates new objects by copying this prototype.
16. **Adapter Pattern**: Allows the interface of an existing class to be used as another interface.
17. **Bridge Pattern**: Decouples an abstraction from its implementation so that the two can vary independently.
18. **Composite Pattern**: Composes objects into tree structures to represent part-whole hierarchies.
19. **Decorator Pattern**: Attaches additional responsibilities to an object dynamically.
20. **Facade Pattern**: Provides a unified interface to a set of interfaces in a subsystem.
21. **Flyweight Pattern**: Uses sharing to support large numbers of fine-grained objects efficiently.
22. **Proxy Pattern**: Provides a surrogate or placeholder for another object to control access to it.
23. **Chain of Responsibility Pattern**: Avoids coupling the sender of a request to its receiver by giving more than one object a chance to handle the request.
24. **Command Pattern**: Encapsulates a request as an object, thereby letting you parameterize clients with different requests.
25. **Interpreter Pattern**: Given a language, defines a representation for its grammar along with an interpreter that uses the representation to interpret sentences in the language.
26. **Iterator Pattern**: Provides a way to access the elements of an aggregate object sequentially without exposing its underlying representation.
27. **Mediator Pattern**: Defines an object that encapsulates how a set of objects interact.
28. **Memento Pattern**: Without violating encapsulation, captures and externalizes an object's internal state so that the object can be restored to this state later.
29. **Observer Pattern**: Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.
30. **State Pattern**: Allows an object to alter its behavior when its internal state changes.
31. **Strategy Pattern**: Defines a family of algorithms, encapsulates each one, and makes them interchangeable.
32. **Template Method Pattern**: Defines the skeleton of an algorithm in an operation, deferring some steps to subclasses.
33. **Visitor Pattern**: Represents an operation to be performed on the elements of an object structure.
34. **Unit Testing**: Testing individual units or components of a software.
35. **Integration Testing**: Testing groups of related modules to ensure they work together correctly.
36. **System Testing**: Testing the complete and integrated software product.
37. **Acceptance Testing**: Testing conducted to determine if the requirements of a specification or contract are met.
38. **Regression Testing**: Re-running functional and non-functional tests to ensure that previously developed and tested software still performs after a change.
39. **Test-Driven Development (TDD)**: A software development process relying on software requirements being converted to test cases before software is fully developed.
40. **Behavior-Driven Development (BDD)**: An extension of TDD that focuses on the behavioral requirement of software.
41. **Continuous Integration (CI)**: The practice of merging all developers' working copies to a shared mainline several times a day.
42. **Continuous Deployment (CD)**: A software engineering approach in which software functionalities are delivered frequently through automated deployments.
43. **Version Control System (VCS)**: A system that records changes to a file or set of files over time so that you can recall specific versions later.
44. **Git**: A distributed version control system for tracking changes in source code during software development.
45. **Branching**: The process of creating a separate line of development in a version control system.
46. **Merging**: The process of combining changes from one branch into another.
47. **Rebasing**: The process of moving or combining a sequence of commits to a new base commit.
48. **Pull Request (PR)**: A method of submitting contributions to an open-source project or onto a specific branch.
49. **Agile Software Development**: An approach to software development under which requirements and solutions evolve through the collaborative effort of self-organizing and cross-functional teams.
50. **Scrum**: An agile framework for managing knowledge work, with an emphasis on software development.
51. **Kanban**: A method for managing the creation of products with an emphasis on continual delivery while not overburdening the development team.
52. **Sprint**: A set period of time during which specific work has to be completed and made ready for review.
53. **Epics**: Large bodies of work that can be broken down into a number of smaller tasks (called stories).
54. **User Stories**: Short, simple descriptions of a feature told from the perspective of the person who desires the new capability.
55. **Refactoring**: The process of restructuring existing computer code—changing the factoring—without changing its external behavior.
56. **Technical Debt**: The implied cost of additional rework caused by choosing an easy solution now instead of using a better approach that would take longer.
57. **Code Review**: A systematic examination of computer source code.
58. **Static Code Analysis**: The analysis of computer software that is performed without actually executing programs.
59. **Dynamic Code Analysis**: The analysis of computer software that is performed by executing programs on a real or virtual processor.
60. **Microservices Architecture**: An architectural style that structures an application as a collection of services that are highly maintainable and testable.
61. **Monolithic Architecture**: A traditional unified model for the design of a software program.
62. **Serverless Computing**: A cloud-computing execution model in which the cloud provider allocates machine resources on demand.
63. **Docker**: A set of platform as a service products that use OS-level virtualization to deliver software in packages called containers.
64. **Kubernetes**: An open-source container-orchestration system for automating computer application deployment, scaling, and management.
65. **API (Application Programming Interface)**: A set of functions and procedures allowing the creation of applications that access the features or data of an operating system, application, or other service.
66. **REST (Representational State Transfer)**: An architectural style for providing standards between computer systems on the web.
67. **GraphQL**: A data query and manipulation language for APIs, and a runtime for fulfilling queries with existing data.
68. **gRPC**: A high-performance, open-source universal RPC framework.
69. **Webhooks**: A way for an app to provide other applications with real-time information.
70. **Authentication**: The process of verifying the identity of a user or process.
71. **Authorization**: The process of specifying access rights/privileges to resources.
72. **JWT (JSON Web Token)**: An open standard that defines a compact and self-contained way for securely transmitting information between parties as a JSON object.
73. **OAuth 2.0**: An industry-standard protocol for authorization.
74. **CORS (Cross-Origin Resource Sharing)**: A mechanism that allows many resources on a web page to be requested from another domain outside the domain from which the resource originated.
75. **Database Normalization**: The process of structuring a relational database in accordance with a series of so-called normal forms.
76. **SQL (Structured Query Language)**: A standard language for accessing and manipulating databases.
77. **NoSQL**: A mechanism for storage and retrieval of data that is modeled in means other than the tabular relations used in relational databases.
78. **ACID Properties**: Atomicity, Consistency, Isolation, Durability - a set of properties of database transactions.
79. **BASE Properties**: Basically Available, Soft state, Eventual consistency - an alternative to ACID for NoSQL databases.
80. **Indexing**: A data structure technique to efficiently retrieve records from the database files based on some attributes on which the indexing has been done.
81. **Query Optimization**: A feature of many relational database management systems in which the database system attempts to determine the most efficient way to execute a given query.
82. **Concurrency Control**: A database management systems requirement for ensuring that database transactions are performed concurrently without violating the data integrity.
83. **Memory Management**: The process of controlling and coordinating computer memory.
84. **Garbage Collection (GC)**: A form of automatic memory management.
85. **Heap Memory**: A region of memory that was used for dynamic memory allocation.
86. **Stack Memory**: A region of memory that is used for local variables and function calls.
87. **Pointer**: A variable whose value is the address of another variable.
88. **Buffer Overflow**: An anomaly where a program, while writing data to a buffer, overruns the buffer's boundary and overwrites adjacent memory locations.
89. **Multithreading**: The ability of a central processing unit (CPU) to provide multiple threads of execution concurrently.
90. **Race Condition**: A phenomenon that may occur inside a critical section where the output of the process is unexpectedly dependent on the sequence or timing of other events.
91. **Deadlock**: A state in which each member of a group is waiting for another member, including itself, to take action.
92. **Mutex (Mutual Exclusion)**: A program object that allows multiple program threads to share the same resource, such as file access, but not simultaneously.
93. **Semaphore**: A variable or abstract data type used to control access to a common resource by multiple processes in a concurrent system.
94. **Event-Driven Programming**: A programming paradigm in which the flow of the program is determined by events such as user actions.
95. **Functional Programming**: A programming paradigm where programs are constructed by applying and composing functions.
96. **Object-Oriented Programming (OOP)**: A programming paradigm based on the concept of "objects", which can contain data and code.
97. **Procedural Programming**: A programming paradigm, derived from imperative programming, based on the concept of the procedure call.
98. **Declarative Programming**: A programming paradigm that expresses the logic of a computation without describing its control flow.
99. **Imperative Programming**: A programming paradigm that uses statements that change a program's state.
100. **Aspect-Oriented Programming (AOP)**: A programming paradigm that aims to increase modularity by allowing the separation of cross-cutting concerns.
